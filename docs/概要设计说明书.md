## 1. 引言

### 1.1 设计目标与范围

本文档是舆情分析平台的概要设计说明书（HLD），依据《舆情分析平台需求规格说明书》（SRS）编写，旨在确定系统的架构风格、技术选型、模块划分、数据架构、接口概要和部署方案。

**在本次设计范围内：**

- 系统整体架构风格与分层设计
- 各功能模块的职责划分与模块间依赖关系
- 技术组件选型及选型理由
- 数据存储方案与数据流转路径
- 模块间通信方式与外部接口清单
- 单机部署拓扑与基础设施需求

**不在本次设计范围内：**

- 具体的 API 端点定义与请求/响应格式（见接口设计说明书）
- 数据库表结构与字段定义（见数据库设计说明书）
- 各模块内部实现细节与代码组织（见详细设计说明书 LLD）
- 前端页面布局与组件树（见详细设计说明书 LLD）
- 测试方案与运维手册

### 1.2 需求概述

本文档上游依据为《舆情分析平台需求规格说明书》，其核心需求可归纳为：

| 模块 | 优先级 | 核心职责 |

## 2. 系统架构

### 2.1 架构风格

采用**模块化单体 + 异步任务**架构：

- **网络采集**作为独立 Python 服务进程运行，通过轮询 PostgreSQL 采集任务表获取任务，与后端完全解耦
- **检索、报告、图谱**三个分析模块通过 LangGraph 编排流程，在 API 后端内以独立线程池执行，与 API 逻辑代码分离
- **API 后端（FastAPI）** 作为统一入口，负责前端请求处理、SSE 推送、业务逻辑和数据层访问

理由：

- 单机部署目标下，微服务架构的运维成本远超收益
- 采集通过 DB 轮询解耦，已经是事实上的独立服务，可独立重启和升级
- 分析模块通过线程池和 LangGraph 边界隔离，未来需要时可平滑拆分为独立服务
- 符合 SRS "单功能应用可独立启停"的语义，同时避免子进程管理的复杂性

**各层职责说明：**

| 层 | 职责 | 关键约束 |
|----|------|---------|
| **客户端层** | 页面渲染、用户交互、SSE 事件接收与消费 | 不直接访问数据库，不直接调用 LLM |
| **API 网关层** | 请求路由、JWT 认证、参数校验、SSE 端点 | 不含业务逻辑，不访问数据层 |
| **业务逻辑层** | 控制台数据聚合、配置读写、用户管理、采集任务 CRUD | 与分析引擎通过明确的服务函数接口通信 |
| **分析引擎层** | LangGraph 工作流执行（检索→报告→图谱），LLM 调用编排 | 运行于独立线程池，与 API 逻辑代码通过函数调用解耦 |
| **数据层** | 全部业务数据持久化，采集任务队列，LangFuse Token 追踪 | 数据库是系统内模块间唯一的共享状态介质 |

**采集服务与后端的解耦方式：**

API 后端和采集服务不直接通信。API 后端收到用户"启动采集"请求后，在 PostgreSQL 的采集任务表中写入任务记录（状态 = `pending`）。采集服务以固定间隔轮询任务表，捡取待执行任务后开始抓取，完成后将状态和结果写回数据库。API 后端通过读取同一张表获取采集进度和结果，推送给前端。这种方式实现了完全的进程间解耦，采集服务可以独立重启、升级，不影响 API 后端。

#### 3.2.3 网络采集服务模块（独立进程）

| 项目 | 内容 |
|------|------|
| **功能** | 独立运行的 Python 进程，轮询数据库中的采集任务，按来源平台并行抓取舆情数据 |
| **输入** | PostgreSQL 采集任务表中的 `pending` 任务记录 |
| **输出** | 采集数据写入采集数据表；任务状态回写采集任务表 |
| **依赖** | PostgreSQL（唯一依赖；不依赖 API 后端进程） |
| **技术** | Python 3.12，独立 uv 虚拟环境 |

**工作流程：**

```
轮询循环（间隔可配，默认 5s）:
  1. SELECT * FROM collection_tasks WHERE status = 'pending' LIMIT 1 FOR UPDATE
  2. UPDATE task SET status = 'running'
  3. 按 task.source_platforms 并行抓取
  4. INSERT INTO collection_data (task_id, platform, content, ...)
  5. UPDATE task SET status = 'stopped', progress = 100
  6. 异常时 UPDATE task SET status = 'failed'
```

**支持的采集平台：** B 站、微博、小红书、抖音、知乎、头条（SRS 3.2 定义），可按来源开关独立控制。

#### 3.2.4 LangGraph 分析引擎（线程池内执行）

| 项目 | 内容 |
|------|------|
| **功能** | 通过 LangGraph 编排和执行三个分析工作流：主题检索、报告生成、图谱构建 |
| **输入** | 分析请求（主题词、限定条件、报告编号等） |
| **输出** | 结构化分析结果写入 PostgreSQL |
| **依赖** | PostgreSQL（读取采集数据、写入分析结果）、外部 LLM API（OpenAI 兼容）、LangFuse（Token 追踪） |
| **技术** | LangGraph + LangChain，执行于独立线程池，通过 LangFuse callback 上报 Token 消耗 |

三个 LangGraph 工作流：

| 工作流 | 触发条件 | 流程简述 |
|--------|---------|---------|
| **主题检索图** | 用户输入主题词发起检索 | 读取采集数据 → 按主题过滤 → LLM 生成来源摘要 → LLM 提炼观点 → LLM 情绪判断 → 写入分析结果表 |
| **报告生成图** | 基于已有检索结果发起报告生成 | 读取检索结果 → 按 8 章节结构（事件概述/时间线/传播渠道/主要观点/情绪倾向/风险判断/重点证据/结论摘要）逐章生成 → 写入报告表 |
| **图谱构建图** | 报告生成后自动触发 | 读取报告内容 → LLM 抽取实体与关系 → 构建节点/边 → 写入图谱数据表 |

**与业务逻辑的解耦方式：**

分析引擎调度器定义一组 Python async 函数（如 `run_search()`, `run_report()`, `run_graph()`），业务逻辑层仅调用这些函数，不感知 LangGraph 内部实现。函数内部将任务提交到独立线程池，返回任务 ID 供状态跟踪。未来如需拆分，将这些函数替换为远程调用接口即可，业务逻辑层无需改动。

#### 3.2.5 进程管理模块

| 项目 | 内容 |
|------|------|
| **功能** | 系统启动时拉起网络采集服务子进程，监控其运行状态，异常时记录错误并推送 SSE |
| **输入** | 系统启动指令 / 采集服务进程信号 |
| **输出** | 进程生命周期日志，SSE `app_status` 和 `error` 事件 |
| **依赖** | Python subprocess / multiprocessing |

说明：本模块仅管理采集服务这一个子进程。检索、报告、图谱在 API 后端线程池内执行，其状态由分析引擎调度器管理。控制台为纯状态展示，无需实现系统级/应用级的启停控制，此模块只做启动拉起和状态监控。

### 3.3 模块间依赖关系

```
                            ┌─────────────────┐
                            │  控制台前端      │
                            │  (Vue 3 +       │
                            │   Element Plus) │
                            └────────┬────────┘
                                     │ REST + SSE
                                     │ (唯一通信协议)
                                     ▼
                            ┌─────────────────┐
                            │  API 后端        │
                            │  (FastAPI)      │
                            └───┬─────────┬───┘
                                │         │
              ┌─────────────────┘         └─────────────┐
              │ 函数调用 (同进程)                         │
              ▼                                          ▼
    ┌─────────────────┐                     ┌─────────────────┐
    │  分析引擎层       │                     │  LangFuse       │

```

**依赖关系矩阵：**

| 调用方 | 被调用方 | 通信方式 | 方向 |
|--------|---------|---------|------|
| 前端 | API 后端 | HTTP REST + SSE | 同步请求 + 单向推送 |
| API 后端 (业务逻辑) | API 后端 (分析引擎) | Python async 函数调用 | 同进程内调用 |
| API 后端 | PostgreSQL | SQL (psycopg/SQLAlchemy) | 同步/异步查询 |
| API 后端 | LangFuse | HTTP (LangFuse SDK) | 分析引擎上报 Token 数据 |
| 分析引擎 | LLM API | HTTP (OpenAI 兼容) | 通过 LangGraph/LangChain 调用 |
| 网络采集服务 | PostgreSQL | SQL 轮询 + 写入 | 唯一依赖；不连接 API 后端 |

**关键约束：**

- 前端不与数据库、LLM API、LangFuse 直接通信
- 网络采集服务不与 API 后端直接通信——数据库是两者唯一的耦合点
- 分析引擎层不直接接收 HTTP 请求——所有外部请求经过 API 网关层和业务逻辑层

### 4.2 存储方案

#### 4.2.1 PostgreSQL 数据库

作为系统**唯一的持久化存储**，承载全部业务数据和任务队列。

| 数据类别 | 说明 | 存储特点 |
|---------|------|---------|
| **用户与角色** | 登录凭据（加密）、角色标识 | 少量、读写频率低 |
| **系统配置** | 运行参数、平台开关、LLM 参数 | 少量、读多写少、支持热更新 |
| **采集任务** | 任务状态、来源平台选择、创建时间 | 中量、采集服务高频轮询读写 |
| **采集数据** | 原始舆情内容（JSON）、来源平台、抓取时间 | 大量、仅追加、按主题和时间查询 |
| **分析结果** | 主题检索产出（来源摘要、观点提炼、情绪判断） | 中量、按主题和报告编号查询 |
| **报告** | 8 章节结构化报告内容 | 少量、按 report_id 精确查找 |
| **图谱数据** | 节点（实体类型+标识）、边（关系类型+来源引用） | 中量、按 report_id 加载整图 |
| **Token 消耗记录** | 按模块/时间维度的消耗统计（LangFuse 为主，此表为备份摘要） | 持续追加、按模块和时间聚合 |
| **运行日志** | 各模块日志摘要 | 大量、按模块和时间查询 |

存储策略要点：

- 采集数据以 JSON 字段存储，保留原始内容结构，不同平台的字段差异由 JSON schema 的灵活性吸收
- 图谱数据采用邻接表模型（节点表 + 边表），便于按 report_id 切换和按节点/关系类型查询。首版不需要引入图数据库或 PG 插件，查询场景为"按报告编号加载整图"和"按节点/关系类型简单过滤"，邻接表完全满足需求
- 采集任务表充当任务队列，采集服务通过 `SELECT ... WHERE status = 'pending' LIMIT 1 FOR UPDATE` 实现任务获取的互斥

**图谱实体与关系设计：**

实体类型：人物、组织机构、事件、话题、信息来源、平台、观点立场、关键词、时间节点

关系类型：发布、来源平台、涉及事件、属于话题、表达观点、持有立场、关联、演变自、提及、包含关键词、发生时间

首版 P1 交付聚焦核心子集：人物、组织机构、事件、话题、信息来源、平台、观点立场 7 类实体；发布、来源平台、涉及事件、属于话题、表达观点、持有立场、关联 7 种关系。

#### 4.2.2 LangFuse — Token 追踪

LangFuse 作为 LLM 可观测性平台独立部署（开源，Docker Compose），负责：

- **Token 用量追踪**：每次 LLM 调用的输入/输出 Token 数、模型、耗时
- **成本统计**：按模块（检索/报告/图谱）、按时间（日/周/月）聚合消耗
- **配额告警**：达到预警阈值通知，达到硬上限暂停调用

LangFuse **不是数据主存储**——业务数据在 PostgreSQL，LangFuse 仅存储 LLM 调用追踪数据。API 后端通过 LangFuse Python SDK 的 callback 机制上报，对分析引擎的 LangGraph 代码透明。

数据分工：

| 存储 | 存什么 | 查询场景 |
|------|-------|---------|
| PostgreSQL | 业务数据、用户、配置、采集/分析/报告/图谱 | 业务查询、历史回查 |
| LangFuse | LLM 调用追踪、Token 消耗、延迟统计 | 成本分析、配额管理、调试 LLM prompt |
| 本地文件系统 | 图谱导出文件（如需）、采集日志文件 | 文件下载、离线分析 |

#### 4.2.3 缓存策略

首版不做额外缓存层。理由：单机部署且并发用户 ≥50 的情况下，PostgreSQL 的查询缓存和连接池足以满足页面首载 ≤2s 和 API ≤200ms 的性能要求。后续若查询性能出现瓶颈，可在 API 后端引入 Redis 缓存热点数据。

#### 4.2.4 消息队列

首版也不引入专用消息队列。唯一需要异步解耦的场景是采集任务调度，已通过数据库轮询实现；SSE 推送在 FastAPI 进程内通过 asyncio 事件广播完成；分析任务通过线程池异步执行。后续若采集任务量大增，可引入 Redis List 或 PostgreSQL LISTEN/NOTIFY 替代轮询。

### 4.3 SSE 实时推送数据路径

```
┌──────────┐       ┌──────────────────┐       ┌──────────┐
│  数据源   │       │    FastAPI       │       │  前端    │
│          │       │                  │       │          │
│ PostgreSQL│      │ ┌──────────────┐ │       │          │
│ 状态变更  │─────►│ │ SSE 事件总线  │ │─────►│ EventSrc │
│          │ 读取  │ │ (asyncio Q)   │ │ 推送 │ 监听     │
│          │      │ └──────────────┘ │       │          │
│ 线程池   │      │        ▲         │       │ 按消息    │
│ 分析完成  │─────►│        │         │       │ 类型更新  │
│          │回调  │   ┌─────┴──────┐ │       │ UI 组件   │
│ 采集服务  │      │   │ SSE 端点   │ │       │          │
│ 写DB后    │─────►│   │ GET /sse  │◄├───────┤ 自动重连  │
│          │读取  │   └────────────┘ │       │          │
└──────────┘       └──────────────────┘       └──────────┘
```

**消息类型与其触发源：**

| 消息类型 | 触发源 | 推送路径 |
|---------|-------|---------|
| `app_status` | 采集服务状态变化 / 分析任务状态变化 | 数据层变更 → 业务逻辑层检测 → SSE 事件总线 → 前端 |
| `app_output` | 分析任务有新的输出内容 | 分析引擎回调 → 事件总线 → 前端 |
| `forum_log` | 采集服务写入新日志 | 采集服务写 DB → API 读取 → 事件总线 → 前端 |
| `system_status` | 系统或应用状态变更 | 状态服务检测 → 事件总线 → 前端 |
| `graph_ready` | 图谱构建完成 | 分析引擎回调 → 事件总线 → 前端 |
| `error` | 任意模块触发异常 | 异常捕获 → 事件总线 → 前端 |

SSE 连接管理：前端通过 EventSource API 连接 `GET /sse` 端点。断开时前端自动重连（间隔 ≤ 3s，满足 SRS 要求）。服务端通过 asyncio 事件队列实现多客户端广播，每个连接订阅全局事件总线，连接断开自动清理。

### 5.2 API 后端 ↔ 网络采集服务

**通信方式：** 无直接通信。通过 PostgreSQL 采集任务表解耦。

**解耦协议（隐式接口）：**

| 方向 | 字段/表 | 说明 |
|------|--------|------|
| API 后端 → 采集服务 | `collection_tasks` 表 INSERT | 写入新任务，`status = 'pending'` |
| 采集服务 → API 后端 | `collection_tasks` 表 UPDATE | 更新任务状态（`running` / `stopped` / `failed`）和进度 |
| 采集服务 → API 后端 | `collection_data` 表 INSERT | 写入采集到的原始舆情数据 |
| 采集服务 → API 后端 | `collection_logs` 表 INSERT | 写入运行日志，供 API 后端查询和 SSE 推送 |

采集服务的数据库连接配置与 API 后端指向同一个 PostgreSQL 实例即可，无额外的进程间调用。

**通信方式：** HTTP（通过 LangFuse Python SDK），由分析引擎的 LangGraph callback 自动上报，业务代码无需手动调用。

| 接口 | 方向 | 说明 |
|------|------|------|

### 6.2 基础设施需求

**进程清单：**

| 进程 | 端口 | 说明 |
|------|------|------|
| Nginx | 80 / 443 | 反向代理 + Vue SPA 静态资源 |
| FastAPI (uvicorn) | 8000 (仅 lo) | 主后端服务，4 workers |
| 网络采集服务 | 无 | API 后端通过 subprocess 拉起 |
| PostgreSQL | 5432 | 唯一业务数据库 |
| LangFuse | 3000, 3030 | Docker Compose 部署，仅用于 Token 追踪 |

**服务器建议：** 4 核 8 GB 起步，200 GB SSD。业务数据和日志月增长约 10 GB，日志保留 30 天滚动清理。

**依赖软件：** Python 3.12 + uv、PostgreSQL 16+、Nginx、Docker + Compose（仅 LangFuse）、Node.js 20（仅前端构建）。

### 6.3 高可用与容错策略

#### 6.3.1 故障场景与应对

| 故障场景 | 影响范围 | 应对策略 |
|---------|---------|---------|
| **网络采集服务崩溃** | 采集停止，其他功能正常 | 进程管理模块检测异常 → 记录 `failed` 状态 → SSE 推送 `error` → 自动重启（最多 3 次） |
| **单个分析任务异常** | 仅该任务失败，其他任务不受影响 | 线程池内异常捕获 → 标记任务状态 = `failed` → SSE 推送 `error` |
| **FastAPI 进程崩溃** | 全站不可用 | systemd/supervisor 配置 `Restart=always`，自动拉起 |
| **PostgreSQL 不可用** | 全站不可用 | 定期备份（pg_dump / WAL 归档），恢复 RPO ≤ 1h |
| **LangFuse 不可用** | LLM 追踪暂时中断，分析功能不受影响 | 分析引擎降级运行——仅日志记录，LangFuse SDK 失败不阻塞流程 |
| **磁盘写满** | 日志/数据写入失败 | 日志轮转策略（保留最近 30 天），采集数据定期清理策略 |
| **LLM API 超时/不可用** | 检索/报告/图谱暂停 | 分析任务标记 `failed`，前端提示 LLM 服务不可用；不丢失已落盘的采集数据 |

#### 6.3.2 数据备份

| 备份对象 | 方式 | 频率 | 保留 |
|---------|------|------|------|
| PostgreSQL 全量 | `pg_dump` | 每日（凌晨低峰） | 最近 7 天滚动 |
| PostgreSQL WAL | `archive_mode` | 持续归档 | 最近 24h |
| 日志文件 | 日志轮转 + 定期清理 | 每天轮转 | 最近 30 天 |

#### 6.3.3 启动与关机顺序

**启动顺序：**

```
PostgreSQL → LangFuse (Docker) → FastAPI (自动拉起采集服务) → Nginx
```

**关机顺序：**

```
Nginx (stop) → FastAPI (等待进行中任务完成，最长 5min) → 采集服务 (stop) → LangFuse (docker stop) → PostgreSQL (stop)
```

#### 6.3.4 监控与告警

| 监控项 | 方式 | 告警阈值 |
|-------|------|---------|
| 各进程存活 | systemd / supervisor 自动监控 | 进程退出 |
| PostgreSQL 连通性 | FastAPI 健康检查端点 | 连接失败 |
| 磁盘使用率 | 系统 cron 脚本 | > 80% |
| Token 配额使用 | LangFuse Dashboard | 达到预警阈值 |
