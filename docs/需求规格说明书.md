# 舆情分析平台需求规格说明书 (SRS)

---

## 1. 引言

### 1.1 项目背景

舆情分析平台旨在为指定主题提供从数据采集到报告生成的全链路舆情分析能力。平台从公开信息、多媒体内容、内部沉淀数据三个方向并行收集信息，经检索分析和洞察提炼后，汇总生成结构化舆情报告，并通过 Web 控制台提供统一的操作入口与结果展示。

### 1.2 文档目的与范围

本文档定义舆情分析平台的功能需求、非功能需求及数据需求，作为后续概要设计、详细设计、开发实现和测试验收的依据。文档覆盖范围包括：

- Web 控制台的页面功能与交互需求
- 各分析应用（网络采集、主题检索、报告生成、图谱分析）的业务规则
- 实时消息通信的类型与字段规范
- 系统级非功能约束（性能、安全、可用性）

### 1.3 术语与缩写

| 术语/缩写 | 说明 |
|-----------|------|
| 单功能应用 | 系统中可独立启停的功能单元，如网络采集、主题检索、报告生成等 |
| 网络采集 | 从公开网络来源并行抓取舆情数据的过程 |
| 图谱 | 以节点和边形式组织的实体关系网络，用于展示舆情信息之间的关联 |
| SSE | Server-Sent Events，服务端向客户端单向推送事件的通信协议 |
| Token | LLM 调用中的最小计费单元，用于衡量模型消耗 |
| uv | Python 虚拟环境管理工具 |
| 报告编号 (report_id) | 唯一标识一次分析报告的字符串，用于报告和图谱的回查索引 |

## 2. 总体描述

### 2.1 产品视角

舆情分析平台是一个独立的 Web 应用系统，通过控制台模块统一管理多个分析子应用的运行生命周期。系统对外通过浏览器提供人机交互界面，对内通过进程管理模块调度各单功能应用的启停与状态监控。

系统与外部环境的关系：

- **上游数据源**：B 站、微博、小红书、抖音、知乎、头条等社交与内容平台——通过网络采集模块接入
- **LLM 服务**：调用外部大语言模型 API 执行内容分析、洞察生成和报告汇总
- **下游消费者**：操作用户通过 Web 控制台查看分析结果和报告；报告查看人导出结构化报告用于决策

系统不依赖外部用户认证系统，自身维护用户角色与权限控制。

### 2.2 用户角色

| 角色 | 核心职责 | 对应页面 |
|------|---------|---------|
| 操作用户 | 启动和停止单功能应用、发起主题检索、查看输出结果 | 控制台首页、主题检索页、图谱查看页 |
| 报告查看人 | 查看已生成报告、图谱结果、采集日志和来源证据 | 图谱查看页、主题检索页 |
| 系统管理员 | 维护系统配置、查看系统状态、启停整体服务、查看运行日志 | 控制台首页、配置管理页、系统状态页 |

### 2.3 运行环境

**服务端：**

| 项目 | 要求 |
|------|------|
| 操作系统 | Linux (生产) / Windows 10+ (开发) |
| Python 版本 | 3.12 |
| 虚拟环境 | uv 管理 |
| 实时通信 | SSE (Server-Sent Events) |

**客户端：**

| 项目 | 要求 |
|------|------|
| 浏览器 | Chrome 90+, Firefox 90+, Edge 90+ |
| 前端框架 | Vue 3 |

### 2.4 约束与假设

**技术约束：**

- 后端语言限定 Python 3.12，使用 uv 管理虚拟环境
- 前端框架限定 Vue 3
- LLM 调用依赖外部 API（网络可达、密钥有效、额度充足）
- 单功能应用以独立子进程方式运行，进程管理模块需处理进程间通信与状态同步

**业务假设：**

- 每次主题检索的操作时长可接受分钟级等待（受限于 LLM 响应时间和数据采集耗时）
- 用户不会同时发起超过 5 个主题检索任务
- 网络采集的目标站点不要求认证登录
- Token 消耗按 LLM 提供商返回的实际用量为准

**优先级约定：**

- P0：核心链路，MVP 必须交付
- P1：重要功能，首版交付
- P2：增强功能，可迭代交付

## 3. 功能需求

### 3.1 控制台模块（P0）

**功能描述**：控制台模块是用户操作系统的统一入口，提供系统级启停、单功能应用启停、状态查看和最近输出摘要。

**用户故事**：
- 系统管理员通过控制台首页一键启动/停止整个系统
- 操作用户查看所有单功能应用的当前状态，独立启动或停止每个应用
- 操作用户在控制台首页查看每个应用最近一次的输出摘要
- 系统管理员查看最近系统错误信息，快速定位问题

**业务规则**：
- 系统状态流转：`offline → starting → online → shutting_down → offline`
- 单功能应用状态流转：`stopped → starting → running → stopping → stopped`，异常时进入 `failed`
- 系统必须先处于 `online` 状态，才能启动单功能应用
- 系统关闭时，先停止所有正在运行的单功能应用，再关闭系统进程
- 停止应用后，已生成的输出和日志保留，不接收新任务

**前置条件**：系统已部署并完成初始配置，用户已登录且有对应角色权限

**后置条件**：应用状态变更通过 SSE 实时推送至前端

---

### 3.2 网络采集模块（P0）

**功能描述**：从多个社交与内容平台并行抓取舆情数据，按来源平台控制采集范围，是主题检索和报告生成的数据基础。

**用户故事**：
- 操作用户启动/停止网络采集任务，选择目标来源平台（如 B 站、微博、小红书、抖音、知乎、头条等）
- 操作用户查看当前采集进度和运行日志，按日期回溯历史采集日志

**业务规则**：
- 采集状态流转：`idle → running → stopped`，异常时进入 `failed`
- 支持按来源平台控制采集开关，可独立启用或禁用特定平台
- 停止采集后，已抓取数据保留，不再发起新请求

**前置条件**：系统在线、目标平台网络可达

**后置条件**：采集数据落盘供下游消费，进度通过 SSE 推送

---

### 3.3 主题检索模块（P0）

**功能描述**：基于网络采集的数据，用户输入主题词和检索条件发起分析，返回包含来源摘要、观点提炼和情绪判断的结构化分析结果。

**用户故事**：
- 操作用户输入主题词、关键词或检索条件，发起主题分析
- 操作用户选择限定来源类型范围（按平台或内容类型过滤）
- 操作用户查看分析输出结果，包括来源摘要和关键发现

**业务规则**：
- 每次主题检索必须保存：主题词、执行时间、来源摘要、分析输出
- 检索结果中的每条结论必须可追溯到具体来源内容
- 支持选择来源类型范围后再检索

**前置条件**：系统在线、网络采集模块处于 `running` 状态

**后置条件**：检索结果持久化保存，可回查

---

### 3.4 报告生成模块（P0）

**功能描述**：汇总主题检索结果，按固定结构生成舆情分析报告，支持按报告编号回查历史报告。

**用户故事**：
- 操作用户发起报告生成，基于指定主题的检索结果汇总输出结构化报告
- 报告查看人按报告编号查阅历史报告
- 报告查看人从报告结论追溯到来源证据

**业务规则**：
- 报告输出必须包含以下 8 个章节：事件概述、时间线、传播渠道、主要观点、情绪倾向、风险判断、重点证据、结论摘要
- 报告结论必须可追溯到来源内容、采集记录或图谱节点

**前置条件**：至少完成一次主题检索

**后置条件**：报告持久化，可被图谱模块和报告回查引用

---

### 3.5 图谱分析模块（P1）

**功能描述**：将舆情数据中的实体及关系以节点-边图谱形式可视化展示，支持节点查询、关系检索和按报告切换。

**用户故事**：
- 报告查看人查看最新图谱或指定报告编号的图谱
- 报告查看人按节点或关系条件查询图谱中的特定实体和关联

**业务规则**：
- 图谱数据至少保存：节点标识、节点类型、关系类型、来源引用
- 支持通过 `/graph-viewer` 和 `/graph-viewer/{report_id}` 进入图谱查看页

**前置条件**：已生成至少一份报告

**后置条件**：图谱数据可被报告模块回溯引用

---

### 3.6 系统状态模块（P1）

**功能描述**：提供整体服务状态、各应用运行状态、最近错误信息和资源运行概况的统一展示面板。

**用户故事**：
- 系统管理员查看整体服务状态和各单功能应用的运行状态
- 系统管理员查看最近错误信息，快速定位异常模块
- 系统管理员查看 CPU、内存等资源运行概况

**业务规则**：
- 系统状态包含：`offline`、`starting`、`online`、`shutting_down`
- 错误信息至少包含：模块名称、错误消息、发生时间
- 状态数据通过 SSE 实时推送更新

**前置条件**：系统已启动

**后置条件**：无

---

### 3.7 配置管理模块（P2）

**功能描述**：提供系统配置、采集配置和运行参数的集中管理，支持读取、修改和即时生效。

**用户故事**：
- 系统管理员查看当前系统配置和运行参数
- 系统管理员修改配置并保存，修改即时生效无需重启

**业务规则**：
- 配置更新后通过热更新机制即时生效
- 非法配置值应拒绝保存并提示错误

**前置条件**：用户具有系统管理员权限

**后置条件**：配置变更可追溯

---

### 3.8 实时消息模块（P0）

**功能描述**：通过 SSE 协议从服务端向客户端单向推送实时状态和事件消息，无需客户端主动轮询。

**用户故事**：
- 前端页面接收各应用状态变更、采集日志流、系统状态变更和错误通知，即时刷新界面

**消息类型及字段定义**：

| 消息类型 | 必含字段 |
|---------|---------|
| `app_status` | `app_name`、`status` |
| `app_output` | `app_name`、`output_text` |
| `forum_log` | `message_text`、`task_status` |
| `system_status` | `system_status`、`running_apps` |
| `graph_ready` | `report_id`、`graph_path` |
| `error` | `module_name`、`error_message` |

**业务规则**：
- 采用 SSE 单向推送，客户端不通过此通道向服务端发送消息
- 连接断开时客户端自动重连

**前置条件**：系统处于 `online` 状态

**后置条件**：无

---

### 3.9 Token 消耗管理模块（P1）

**功能描述**：统计各 LLM 调用环节的 Token 消耗量，提供配额管理和消耗预警。

**用户故事**：
- 系统管理员查看各模块的 Token 用量统计
- 系统管理员设置 Token 配额上限
- 消耗接近配额上限时触发告警通知

**业务规则**：
- 按模块和按时间维度统计 Token 消耗（支持日/周/月粒度）
- 配额分为预警阈值和硬上限两档，达到预警阈值时触发通知，达到硬上限时暂停 LLM 调用
- 消耗记录持久化，支持历史查询

**前置条件**：系统已接入 LLM API

**后置条件**：消耗数据写入持久存储

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 要求 |
|------|------|
| 页面首次加载时间 | ≤ 2s |
| API 响应时间（非检索类） | ≤ 200ms (P95) |
| 主题检索任务响应时间 | ≤ 5min |
| 并发用户数 | ≥ 50 |
| 同时运行的单功能应用数 | ≥ 10 |
| SSE 重连延迟 | ≤ 3s |

### 4.2 安全需求

- 用户需通过账号密码登录后访问控制台，禁止未认证访问
- 角色权限分离：操作用户、报告查看人、系统管理员各有限定的操作范围
- 所有用户输入需做输入校验和注入防护
- LLM API Key 等敏感配置项加密存储，前端不可见

### 4.3 可用性需求

- 系统可用率 ≥ 99.5%
- 单功能应用异常退出时自动标记为 `failed`，不影响其他应用正常运行
- 系统崩溃重启后，已持久化的报告、图谱数据不丢失

### 4.4 可维护性需求

- 每个单功能应用独立记录运行日志，便于隔离排查
- 配置修改支持热更新，无需重启系统

## 5. 数据需求

### 5.1 核心数据实体

| 数据实体 | 说明 |
|---------|------|
| 系统配置 | 运行参数、采集平台开关、LLM 参数等 |
| 采集数据 | 网络采集模块抓取的原始舆情数据，标注来源平台和时间 |
| 分析结果 | 主题检索产出的分析输出，含来源摘要和关键发现 |
| 分析报告 | 按固定结构汇总的结构化舆情报告，支持按编号回查 |
| 图谱数据 | 舆情实体节点与关系边，含节点类型、关系类型和来源引用 |
| Token 消耗记录 | 各模块 LLM 调用消耗统计，支持按模块和时间维度查询 |

### 5.2 数据流转概要

```
网络采集 → 主题检索 → 报告生成 → 图谱分析
    │                      │
    └── 原始数据支撑 ──────┘
```

采集数据是检索分析的基础，检索结果汇总为报告，报告提炼出图谱实体关系，三者形成上下游数据依赖链。
